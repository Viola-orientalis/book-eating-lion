# 프로젝트 개요

# **프로젝트 명 : 책 먹는 사자**

## **1\. 프로젝트 개요**

사용자가 온라인으로 도서를 조회하고 장바구니에 담아 주문한 뒤, 테스트용 가상 카드를 이용해 결제할 수 있는 웹 쇼핑몰을 구축한다.

프론트엔드는 React로 구현하고, 백엔드는 Spring Boot 기반 REST API로 구성한다. 애플리케이션은 AWS EC2의 Docker 컨테이너에서 실행하며, 회원·도서·주문·결제 정보는 Amazon RDS에 저장한다.

도서 표지 이미지와 결제 명세서 파일은 Amazon S3에 저장하고, CloudWatch를 이용하여 EC2와 RDS의 상태를 모니터링한다.

* React와 Spring Boot를 연동한 웹 서비스 구현  
* 회원, 도서, 장바구니, 주문, 결제 도메인 설계  
* 주문·결제·재고 데이터를 하나의 트랜잭션으로 처리  
* AWS VPC 기반 Public·Private 네트워크 구성  
* EC2와 Docker를 활용한 애플리케이션 배포  
* Private Subnet의 RDS와 애플리케이션 연동  
* IAM Role을 이용한 안전한 S3 접근  
* CloudWatch 기반 서버 및 데이터베이스 모니터링  
* 보안·안정성·비용을 고려한 클라우드 인프라 구축

## **2\. 프로젝트 목적**

온라인 서점은 회원, 상품, 장바구니, 주문, 결제, 재고 등 서로 연관된 여러 데이터를 처리해야 하는 대표적인 웹 서비스이다.

본 프로젝트는 단순한 상품 조회 기능을 구현하는 데 그치지 않고, 사용자의 주문 요청이 서버와 데이터베이스, 파일 스토리지를 거쳐 안전하게 처리되는 전체 서비스 흐름을 구현하는 것을 목적으로 한다.

특히 여러 사용자가 동시에 도서를 주문하는 상황에서도 재고가 음수가 되거나, 결제는 완료되었지만 주문 상태가 변경되지 않는 데이터 불일치가 발생하지 않도록 주문·결제·재고 변경을 하나의 트랜잭션으로 처리한다. 이를 통해 실제 쇼핑몰 서비스에서 중요한 데이터 정합성, 예외 처리, 동시성 제어 방식을 사용한다.

또한 개발한 애플리케이션을 로컬 환경에서만 실행하는 것이 아니라 AWS 클라우드 환경에 직접 배포하여 서버, 네트워크, 데이터베이스, 스토리지, 권한, 모니터링이 연결되는 과정을 경험한다. 이를 통해 애플리케이션 개발뿐 아니라 서비스를 실제로 운영하기 위해 필요한 클라우드 인프라 구성 역량을 함께 확보한다.

### **사용 기술 스택**

#### **React**

사용자가 도서 목록 조회, 장바구니 관리, 주문, 결제, 주문 내역 확인 등의 기능을 화면에서 직관적으로 이용할 수 있도록 프론트엔드를 React로 구현한다.

React의 컴포넌트 기반 구조를 활용하여 도서 카드, 장바구니 항목, 주문 목록 등의 반복되는 UI를 재사용하고, 화면 상태와 서버 데이터를 효율적으로 관리한다. 또한 Spring Boot와 REST API 방식으로 통신하여 프론트엔드와 백엔드의 역할을 명확하게 분리한다.

#### **Spring Boot**

회원 인증, 도서 관리, 주문 생성, 결제 승인, 재고 차감과 같은 핵심 비즈니스 로직을 처리하기 위해 Spring Boot를 사용한다.

Controller, Service, Repository 또는 Mapper 계층을 분리하여 기능별 책임을 명확하게 구성하고, `@Transactional`을 이용해 주문·결제·재고 변경 과정에서 오류가 발생하면 전체 작업이 롤백되도록 구현한다.

Spring Security와 BCrypt를 적용하여 사용자 인증과 권한을 관리하고, 비밀번호가 평문으로 저장되지 않도록 한다.

#### **Amazon RDS**

회원, 도서, 장바구니, 주문, 주문 상품, 가상 카드, 결제 내역처럼 관계가 명확한 정형 데이터를 저장하기 위해 Amazon RDS MySQL을 사용한다.

RDS를 Private Subnet에 배치하여 외부 인터넷에서 직접 접근하지 못하도록 하고, EC2의 애플리케이션 서버를 통해서만 데이터베이스에 접근하도록 구성한다. 이를 통해 실제 서비스에서 사용하는 데이터베이스 보호 구조를 학습한다.

#### **Amazon S3**

도서 표지 이미지와 결제 명세서처럼 데이터베이스 테이블에 직접 저장하기 부적절한 파일을 관리하기 위해 Amazon S3를 사용한다.

파일 자체는 S3에 저장하고, RDS에는 파일명과 S3 Object Key 등의 메타데이터만 저장한다. 이를 통해 정형 데이터와 비정형 파일을 목적에 맞게 분리하고, 파일 저장에 따른 데이터베이스 용량과 백업 부담을 줄인다.

EC2에서 S3에 접근할 때는 Access Key를 코드에 저장하지 않고 IAM Role을 사용하여 인증정보 노출 위험을 줄인다.

#### **Amazon EC2**

React와 Spring Boot 애플리케이션을 외부 사용자가 접속할 수 있는 서버 환경에 배포하기 위해 Amazon EC2를 사용한다.

EC2를 Public Subnet에 배치하고, 사용자의 요청을 받아 Spring Boot와 RDS, S3로 전달하는 애플리케이션 서버 역할을 수행하도록 한다.

#### **Docker와 Docker Compose**

개발자의 로컬 환경과 EC2 배포 환경의 차이로 발생할 수 있는 실행 오류를 줄이기 위해 애플리케이션을 Docker 이미지로 구성한다.

React·Nginx 컨테이너와 Spring Boot 컨테이너를 분리하여 각 서비스의 실행 환경을 독립적으로 관리하고, Docker Compose를 이용해 여러 컨테이너를 한 번에 실행할 수 있도록 한다.

이를 통해 애플리케이션 실행 환경의 일관성을 확보하고 배포 과정을 단순화한다.

#### **Amazon VPC**

외부 사용자가 접근하는 애플리케이션 서버와 외부에 노출되면 안 되는 데이터베이스를 분리하기 위해 VPC를 구성한다.

EC2는 Public Subnet에 배치하고, RDS는 서로 다른 가용 영역의 Private Subnet에 배치한다. Security Group을 이용하여 RDS는 EC2에서 전달되는 MySQL 요청만 허용한다.

이를 통해 Public 영역과 Private 영역을 분리하고, 서비스 간 필요한 통신만 허용하는 네트워크 보안 구조를 구현한다.

#### **Amazon CloudWatch**

서비스 배포 후 서버와 데이터베이스가 정상적으로 동작하는지 확인하기 위해 CloudWatch를 사용한다.

EC2의 CPU 사용률과 네트워크 사용량, RDS의 CPU 사용률, 데이터베이스 연결 수, 남은 저장 공간 등을 확인한다. 또한 부하 테스트 전후의 지표를 비교하여 병목 구간과 장애 원인을 분석한다.

### **최종 목표**

본 프로젝트의 최종 목표는 사용자가 React 기반 서점 웹사이트에서 도서를 조회하고 주문·결제를 수행하면, EC2의 Docker 애플리케이션이 요청을 처리하고, 주문 및 결제 데이터는 Private RDS에 저장하며, 도서 이미지와 결제 명세서는 S3에 저장하는 전체 클라우드 서비스 흐름을 완성하는 것이다.

이를 통해 웹 애플리케이션 개발, 데이터 정합성 관리, 컨테이너 배포, AWS 네트워크 구성, 데이터베이스 보안, 파일 스토리지 연동, 모니터링까지 하나의 프로젝트에서 종합적으로 경험하는 것을 목적으로 한다.

# 기능 요구사항

# **기능적 요구사항**

## **1\. 회원 및 인증 관리**

### **회원가입**

* 사용자는 아이디, 비밀번호, 이름을 입력하여 회원가입한다.  
* 아이디는 중복될 수 없다.  
* 비밀번호는 BCrypt 등을 이용해 단방향 암호화한 후 저장한다.  
* 신규 회원의 기본 권한은 `USER`로 설정한다. (관리자를 추가할 수도 있기 때문)

### **로그인**

* 사용자는 아이디와 비밀번호로 로그인한다.  
* 로그인 성공 시 서버는 세션 또는 JWT를 발급한다.  
* 로그인하지 않은 사용자는 장바구니, 주문, 결제 기능을 이용할 수 없다.  
* 인증된 사용자는 자신의 주문과 결제 정보만 조회할 수 있다.

## **2\. 도서 관리**

### **도서 목록 조회**

* 모든 사용자는 판매 중인 도서 목록을 조회할 수 있다.  
* 도서명, 저자, 가격, 재고, 카테고리, 표지 이미지를 표시한다.  
* 도서명 또는 저자를 이용한 검색 기능을 제공한다.  
* 카테고리별 도서 조회 기능을 제공한다.

### **도서 상세 조회**

도서 상세 화면에 다음 정보를 표시한다.

* 도서명  
* 저자  
* 출판사  
* ISBN  
* 가격  
* 재고  
* 카테고리  
* 도서 설명  
* 표지 이미지

### **도서 등록 및 수정**

* 도서 표지 이미지는 S3에 업로드한다.  
* RDS에는 이미지 파일 자체가 아니라 S3 Object Key 또는 이미지 URL을 저장한다.

### **도서 삭제 및 판매 상태 관리 (필수 구현 X)**

* 기존 주문에 포함된 도서는 물리적으로 삭제하기보다 판매 중지 처리하는 것을 우선한다.  
* 판매 중지된 도서는 새로운 주문에 포함할 수 없다.

## **3\. 장바구니 관리**

### **장바구니 추가**

* 사용자는 도서 상세 화면에서 수량을 선택해 장바구니에 추가할 수 있다.  
* 장바구니 수량은 1개 이상이어야 한다.  
* 현재 재고보다 많은 수량은 선택할 수 없다.

### **장바구니 조회 및 변경**

* 사용자는 장바구니에 담긴 도서 목록을 조회할 수 있다.  
* 도서별 수량과 금액을 표시한다.  
* 사용자는 장바구니 수량을 변경하거나 항목을 삭제할 수 있다.  
* 전체 주문 예정 금액을 계산해 표시한다.

## **4\. 주문 관리**

### **주문 생성**

* 사용자가 구매를 진행하면 주문 정보를 생성한다.  
* 주문 초기 상태는 `PENDING_PAYMENT`로 설정한다.  
* 주문과 주문 상품 정보를 분리하여 저장한다.  
* 주문 당시 도서명, 가격, 수량을 주문 상품 정보에 저장한다.

### **주문 금액 계산**

* 주문 총액은 각 도서의 주문 당시 가격과 수량을 기준으로 계산한다.  
* 프론트엔드에서 전달한 총액을 그대로 신뢰하지 않는다.  
* 서버가 RDS의 도서 가격을 다시 조회하여 최종 금액을 계산한다.

### **주문 상태 관리**

주문 상태는 다음과 같이 관리한다.

* `PENDING_PAYMENT`: 결제 대기  
* `PAID`: 결제 완료  
* `PAYMENT_FAILED`: 결제 실패  
* `CANCELLED`: 주문 취소

## **5\. 가상 카드 관리**

### **가상 카드 발급**

* 회원은 프로젝트용 가상 카드를 발급받을 수 있다.  
* 카드 발급 시 이용 한도와 카드 상태를 설정한다.  
* 실제 카드번호와 CVV는 저장하지 않는다.  
* UUID 기반 카드 토큰과 마스킹 카드번호만 RDS에 저장한다.

### **카드 정보 조회**

사용자는 다음 카드 정보를 조회할 수 있다.

* 마스킹 카드번호  
* 카드 상태  
* 총 이용 한도  
* 현재 사용 금액  
* 남은 이용 가능 금액

### **카드 상태 관리**

카드 상태는 다음과 같이 구분한다.

* `ACTIVE`: 정상  
* `SUSPENDED`: 일시 정지  
* `CLOSED`: 해지

정지 또는 해지 상태의 카드는 결제에 사용할 수 없다.

## **6\. 결제 관리**

### **결제 요청**

사용자는 결제 대기 상태의 주문과 사용할 가상 카드를 선택하여 결제를 요청한다.

서버는 다음 항목을 검증한다. (몇 개만 선택하면 될 듯 합니다.)

* 주문 존재 여부  
* 주문 소유자 확인  
* 주문 상태 확인  
* 카드 존재 여부  
* 카드 소유자 확인  
* 카드 상태 확인  
* 카드 잔여 한도 확인  
* 도서 판매 상태 확인  
* 도서 재고 확인  
* 중복 결제 요청 여부 확인

### **결제 승인**

검증을 통과하면 다음 작업을 수행한다.

1. 도서 재고 차감  
2. 카드 사용 금액 증가  
3. 결제 성공 내역 저장  
4. 주문 상태를 `PAID`로 변경  
5. 승인번호 생성  
6. 결제 일시 저장

위 작업은 하나의 데이터베이스 트랜잭션으로 처리한다.

작업 중 하나라도 실패하면 모든 변경 사항을 롤백한다.

### **결제 거절**

다음 조건에 해당하면 결제를 거절한다. (구현 내용에 따라 선택하면 될 듯 합니다.)

* 정지 또는 해지 카드  
* 카드 한도 초과  
* 재고 부족  
* 판매 중지 도서  
* 이미 결제된 주문  
* 다른 사용자의 카드  
* 잘못된 결제 금액  
* 중복 결제 요청

결제 거절 시 카드 사용 금액과 재고를 변경하지 않는다.

결제 내역에는 거절 상태와 거절 사유를 저장한다.

### **결제 취소**

* 사용자는 자신이 결제한 주문을 취소할 수 있다.  
* 승인된 결제만 취소할 수 있다.  
* 이미 취소된 결제는 다시 취소할 수 없다.

취소 시 다음 작업을 수행한다.

1. 카드 사용 금액 복구  
2. 도서 재고 복구  
3. 결제 상태를 `CANCELLED`로 변경  
4. 주문 상태를 `CANCELLED`로 변경  
5. 취소 일시 저장

위 작업도 하나의 데이터베이스 트랜잭션으로 처리한다.

## **7\. 주문 및 결제 내역**

### **주문 내역 조회**

사용자는 자신의 주문 내역을 시간순으로 조회할 수 있다.

조회 항목은 다음과 같다.

* 주문번호  
* 주문 도서  
* 주문 수량  
* 주문 총액  
* 주문 상태  
* 주문 일시

### **결제 내역 조회**

사용자는 자신의 결제 내역을 조회할 수 있다.

조회 항목은 다음과 같다.

* 결제번호  
* 승인번호  
* 주문번호  
* 마스킹 카드번호  
* 결제 금액  
* 결제 상태  
* 승인 또는 거절 사유  
* 결제 일시  
* 취소 일시

## **8\. S3 파일 관리**

### **도서 표지 이미지 저장**

* 관리자가 업로드한 도서 표지 이미지를 S3에 저장한다.  
* S3 Public Access Block을 활성화한다.  
* 필요한 경우 애플리케이션을 통해 이미지를 제공하거나 제한된 URL을 생성한다.

### **결제 명세서 생성**

결제 성공 시 다음 정보를 포함한 JSON, CSV 또는 PDF 명세서를 생성한다.

* 사용자 이름  
* 주문번호  
* 승인번호  
* 구매 도서 목록  
* 결제 금액  
* 결제 일시  
* 마스킹 카드번호

### **명세서 업로드 및 조회**

* 생성된 명세서 파일을 S3에 업로드한다.  
* RDS에는 명세서 ID, 주문 ID, S3 Object Key, 생성 일시를 저장한다.  
* 사용자는 마이페이지에서 명세서를 조회하거나 다운로드할 수 있다.  
* EC2는 Access Key 대신 IAM Role을 이용해 S3에 접근한다.

## **9\. 서비스 상태 확인**

### **헬스 체크**

* Spring Boot 애플리케이션의 실행 상태를 확인할 수 있는 `/health` 또는 Actuator 엔드포인트를 제공한다.  
* RDS 연결 상태를 확인할 수 있어야 한다.  
* Docker 컨테이너 실행 상태와 로그를 확인할 수 있어야 한다.

# 비기능 요구사항

# **비기능적 요구사항**

## **보안**

* 비밀번호는 BCrypt로 암호화하여 저장한다.  
* 실제 카드번호, CVV, PIN은 저장하지 않는다.  
* 카드번호는 마스킹하여 표시한다.  
* 인증정보와 DB 비밀번호를 소스코드에 작성하지 않는다.  
* `.env`, Key, Token, Secret을 GitHub에 업로드하지 않는다.  
* 로그에 비밀번호나 카드 토큰 전체 값을 출력하지 않는다.  
* S3 접근은 EC2 IAM Role을 이용한다.  
* RDS는 외부 인터넷에 공개하지 않는다.

## **데이터 정합성**

* 결제 승인 시 재고, 카드 사용 금액, 결제 내역, 주문 상태를 하나의 트랜잭션으로 처리한다.  
* 결제 취소 시 카드 한도, 재고, 결제 상태, 주문 상태를 하나의 트랜잭션으로 처리한다.  
* 오류 발생 시 전체 작업을 롤백한다.

## **동시성**

* 여러 사용자가 같은 도서를 동시에 구매해도 재고가 음수가 되지 않아야 한다.  
* 동일 카드에 결제 요청이 동시에 발생해도 한도를 초과해서 승인되지 않아야 한다.  
* 비관적 잠금 또는 조건부 업데이트를 이용해 동시성 문제를 처리한다.

## **성능**

* 일반적인 도서 목록과 상세 조회 요청이 정상적으로 응답해야 한다.  
* k6 등을 이용하여 도서 조회와 주문·결제 API에 부하 테스트를 수행한다.  
* 부하 테스트 전후 EC2와 RDS 지표를 비교한다.

## **모니터링**

CloudWatch에서 다음 항목을 확인한다.

* EC2 CPU 사용률  
* EC2 네트워크 사용량  
* EC2 상태 검사  
* RDS CPU 사용률  
* RDS Database Connection  
* RDS 남은 저장 공간  
* RDS 읽기·쓰기 지연시간

## **유지보수성**

* React 프론트엔드와 Spring Boot 백엔드를 분리한다.  
* Controller, Service, Repository 또는 Mapper 계층을 구분한다.  
* 기능별 도메인을 분리한다.  
* API 요청·응답 DTO를 사용한다.  
* 오류 코드와 응답 형식을 통일한다.

# 인프라 요구사항

# **AWS 인프라 요구사항**

## **VPC 및 네트워크**

* 사용자 정의 VPC를 생성한다.  
* Public Subnet을 1개 이상 생성한다.  
* 서로 다른 가용 영역에 Private Subnet을 2개 생성한다.  
* Internet Gateway를 생성하고 VPC에 연결한다.  
* Public Route Table과 Private Route Table을 구성한다.

## **EC2 및 Docker**

* Public Subnet에 EC2 인스턴스를 생성한다.  
* EC2에 Docker와 Docker Compose를 설치한다.  
* React·Nginx와 Spring Boot를 컨테이너로 실행한다.  
* 외부 브라우저에서 서점 쇼핑몰에 접속할 수 있어야 한다.  
* Dockerfile과 설정 파일은 GitHub에서 관리한다.

## **Amazon RDS**

* RDS MySQL을 Private Subnet에 배치한다.  
* 서로 다른 가용 영역의 Private Subnet으로 DB Subnet Group을 구성한다.  
* RDS Public Access를 비활성화한다.  
* RDS Security Group의 MySQL 포트는 EC2 Security Group에서만 접근을 허용한다.  
* RDS 자동 백업과 Storage Auto Scaling을 설정한다.

## **Amazon S3**

* 도서 이미지와 결제 명세서 저장용 S3 Bucket을 생성한다.  
* Public Access Block을 활성화한다.  
* 서버 측 암호화를 활성화한다.  
* EC2 IAM Role을 통해 파일 업로드와 조회를 수행한다.

## **Amazon EBS**

* EC2에 연결된 EBS의 역할과 용량을 확인한다.  
* EBS에 Docker 이미지, 컨테이너 파일, 애플리케이션 로그 등이 저장되는 구조를 확인한다.  
* EBS 정보를 리소스 구성 명세에 기록한다.

## **CloudWatch**

* EC2와 RDS의 주요 지표를 확인한다.  
* 모니터링 결과를 캡처하고 분석한다.  
* 컨테이너 오류와 해결 과정을 트러블슈팅 문서에 기록한다.

## **태그와 비용 관리**

* AWS 주요 리소스에 `Team=team3` 태그를 설정한다.  
* 비용이 발생하는 EC2, RDS, EBS, S3 등의 리소스를 확인한다.  
* 발표와 시연 종료 후 불필요한 리소스를 정리한다.
